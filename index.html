About me

I worked at SEGA from mid-2017 until April 2024, beginning as a contract employee and eventually managing the web team. Despite the team consisting of just three members, we accomplished a remarkable amount of work as the number of titles released grew each year, with roles expanding including into CRM. I operated with the efficiency and speed typical of an agency, often juggling two to three projects while managing all of the often daily unexpected requests. As a team manager, I also ran stand-ups, created work-back schedules, assigned projects, and collaborated with various teams to discuss upcoming web projects and campaign expectations. Despite the overwhelming demands, I consistently completed my projects on time, if not early.

As a team player, I was always ready to support both the web and design teams, offering help whenever needed. I collaborated with all levels of management and communicated with external partners, including Nintendo, Sony, Microsoft, the International Olympic Committee, Blue Planet Software, and others. My role also involved working with other SEGA divisions, such as SEGA of Europe and the Sonic team.

Beyond my work on numerous websites, I contributed to several key initiatives. I helped develop a template to streamline site development, designed and coded a universal toolbar now featured across most SEGA and Atlus sites, and created a Webpack bundler to support the latest JavaScript, SCSS, and CSS files. Additionally, I worked closely with the Data Protection Officer to ensure GDPR and CCPA compliance, introduced A/B testing for performance improvements, and built high-traffic sites for popular titles featuring Sonic, Mario, and Tetris.

Since then, I have transitioned to freelance work while expanding my skill set. My passion for coding, continues to drive me forward, and I remain committed to continuous learning through online courses and staying up-to-date with the latest developments in the field.

Mario And Sonic

at the olympic games

Visit website

Technologies Used: JavaScript, jQuery, HTML, CSS

Mario and Sonic at the Olympic Games is likely the most prolific website I've worked on, with more visitors than all the other sites I've been involved with combined. Working on a project that included both SEGA and Nintendo was incredibly special for me. I designed the site to emphasize the party-game aspect, incorporating small looping MP4 clips for each event when hovered over, animated stars in the background, and animated Nintendo Switch controllers to add a touch of fun.

This project was a major global announcement tied to E3Expo and supported more countries than we had ever handled before. To manage this, I developed a JavaScript constructor that routed visitors to the correct page based on their location. The code first attempted to detect the browser language, and if that failed, it called an API to determine the country of origin. This approach helped reduce API call costs while ensuring GDPR compliance by only requesting the country location. With the country determined, I used six different arrays containing country codes to construct the correct URL for each visitor.

One of the biggest challenges was the sheer number of videos on the site. Even though they were small, they caused significant issues. My solution was to lazy load the videos after the site loaded, which worked well on desktop browsers.

Battle Match

One Piece themed memory match game

See Application See The Code

One Piece Battle Match is a spin off of the classic memory match game. As a player, you still make matches but, you are battling a computer based opponent that can damage you as well. You don’t always have to make all the matches to actually defeat your opponent but, it is the main purpose of the game. The game allows you to chose your hero based on several of the Straw Hat crew member and, each character has their own set of cards depicting their fighting styles with unique actions based on their actual fighting styles.

Each turn you flip two cards and, if they match the action on the card occurs. If no match is made, a story unfolds in the middle of the screen. Following your turn, the villain takes his action which is shown within the same story window. Some times damage is dealt, other times, it is just flavor text to add to add to the overall theme.

The code is a combination of HTML5, CSS3, jQuery and JavaScript. The health bars and, one of the modals were developed from Bootstrap. The javaScript is written in a procedural format as this was written in our second week of the program and, we were still in the process of learning the basics.

Technologies Used: React, Redux, Hooks, Context

I created a weather app that provides detailed weather data based on user-inputted zip codes. The app displays both hourly and daily weather results, with clickable days that reveal more specific information for the selected date.

This application was developed using React, and I implemented it in four different ways. Though the user interface remains consistent, the underlying code varies. Initially, I built the app using component state and passed props down to child components. I then refactored the application using React with Redux, followed by React with hooks, and finally React with Context.

To add complexity, I integrated two different APIs. First, I used the Google Geocoding API to convert the zip code into latitude and longitude coordinates. If the coordinates are successfully retrieved, the second API, OpenWeather, is called to fetch the weather data. If an invalid zip code is entered, the app displays a warning message instead of calling the weather API. Additionally, I used react-router-dom and createHashHistory to manage routing within the application, allowing users to navigate to specific days' weather reports.

Technologies Used: React, Redux, Hooks, Context

This component mimics the social card interfaces commonly found on platforms like Twitter, Facebook, Pinterest, and Redfin. I deconstructed a typical social card into smaller, reusable parts to create a list that populates with a random amount of incoming data. When the application loads, a call is made to faker.js to retrieve data, with the number of items determined by a randomly generated number.

The card is divided into two main sections: the header and the main content area. The main content is further split into a colored background section and a descriptive text area below it. This modular approach makes it easy to reuse, adjust, and maintain the components.

Additionally, the background color for each card is randomly selected using faker.js. To ensure readability, a check was implemented to replace light colors that would make the white text difficult to read with darker shades.
